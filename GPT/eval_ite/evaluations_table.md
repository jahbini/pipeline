Table: `evaluations` (SQLite, in `runtime.sqlite`)
Producer: `judge_run_ite` (`scripts/eval_ite/judge_run_ite.coffee`)
Schema home: `meta/sqlite.coffee` (search `CREATE TABLE IF NOT EXISTS evaluations`)

## Purpose

One row per pipeline run that reached `judge_run_ite`. Captures the
score, the formula used, key metric breakdowns, an eval-prompts hash
for comparability, a per-step hyperparams snapshot, and the full
`eval_score` artifact as `details_json` for forensics. This is the
table the advice loop (see `/CLAUDE.md`) consumes to compare runs.

## Schema

| column | type | meaning |
|---|---|---|
| `run_id` | TEXT PRIMARY KEY | FK to `runs.run_id`; UUID generated by the runner at startup |
| `pipeline` | TEXT | recipe name (e.g. `eval_ite`, `full_cycle_ite`) — copied from the `runs` row for query convenience |
| `judged_at` | TEXT | ISO timestamp written by `judge_run_ite` |
| `formula` | TEXT | the score formula string (includes `(FALLBACK: ...)` when distinct2 stood in for voice_similarity) |
| `fallback_used` | INTEGER | `1` if voice_similarity was unavailable, `0` if it was used |
| `score_base` | REAL | per-variant scores after the formula; `NULL` if that variant isn't in the run |
| `score_with_adapter` | REAL | "" |
| `delta` | REAL | `score_with_adapter - score_base`; `NULL` when only one variant |
| `verdict` | TEXT | `'better'`/`'worse'`/`'unchanged'`/`'single-variant'` (see judge_run_ite for thresholds) |
| `voice_cosine_base` | REAL | from `voice_similarity.by_variant.base.cosine_mean`; `NULL` when `fallback_used = 1` |
| `voice_cosine_with_adapter` | REAL | "" |
| `mem_sub_rate_base` | REAL | from `ablation_summary.by_variant.base.mem_sub_rate` |
| `mem_sub_rate_with_adapter` | REAL | "" |
| `distinct2_base` | REAL | from `ablation_summary.by_variant.base.distinct2_mean` |
| `distinct2_with_adapter` | REAL | "" |
| `eval_prompts_hash` | TEXT | sha1 hex of `JSON.stringify(eval_prompts)` (NULL if no prompts) — use this to filter "comparable" runs |
| `eval_prompts_count` | INTEGER | `eval_prompts.length` |
| `hyperparams_json` | TEXT | JSON: `{stepName: <params/<stepName>.yaml contents>, ...}` — snapshot of the steps named in `judge_run_ite.hyperparam_steps` |
| `details_json` | TEXT | JSON: the full `eval_score` artifact (for forensics; `score_by_variant`, `summary`, `voice_similarity`, etc.) |
| `created_at` | TEXT | ISO; equals `judged_at` for fresh writes |

Indexes: `idx_evaluations_judged_at`, `idx_evaluations_pipeline`.

Triggers: INSERT/UPDATE/DELETE all populate `_change_log` so
`/api/sqlite/diff?since=<run_id>` and the agent surface see eval
changes alongside other table activity.

## Meta requests

| request key | direction | body / result |
|---|---|---|
| `evaluationRegister{<run_id>}.json` | write | full row object as built by judge_run_ite. INSERT OR REPLACE. The `run_id` in the body and the curly-braces argument must match |
| `evaluation{<run_id>}.json` | read | one row, with `hyperparams_json` and `details_json` parsed into `hyperparams` and `details` object fields |
| `evaluationHistory.jsonl` | read | all rows ordered by `judged_at DESC`, with JSON columns parsed |
| `evaluationLatest.json` | read | the most recent row (or `null`) |
| `evaluationsByPromptHash{<hash>}.jsonl` | read | every row sharing one `eval_prompts_hash`, ordered `judged_at DESC`; JSON columns parsed. Use this — and ONLY this — to trend a score across runs |
| `trainingHistoryJoinEval.jsonl` | read | every evaluation row joined to the training that was in effect when it was scored (temporal LEFT JOIN to `lora_training_runs` on `finished_at <= judged_at`, most recent wins). Training columns prefixed `training_`. NULL when no training row exists yet. See "Cross-table join" below |

## Producer: judge_run_ite

After making the `eval_score` artifact, the step:

1. Reads the current `run_id` from the memo (`run/current_run_id`,
   published by the runner at startup).
2. Collects hyperparams from each step named in
   `judge_run_ite.hyperparam_steps` (recipe param; default is the four
   steps holding all tunable knobs — `run_lora_train_ite`,
   `generate_ablations_ite`, `oracle_ask_sqlite`,
   `collect_diary_kag_ite`) by reading `params/<stepName>.yaml` from
   the memo. Captures each step's full params blob; that includes the
   `mlx:` sub-block.
3. Computes `eval_prompts_hash` (sha1 of stringified
   `generate_ablations_ite.eval_prompts`).
4. Builds the row object and writes via
   `L.saveThis "evaluationRegister{#{runID}}.json", row`.

Best-effort: a missing `run_id`, missing meta, or any write error
logs to stderr but does NOT fail the step. The `eval_score` artifact
remains the source of truth.

## Reading patterns (for the advice loop)

```coffee
# Score and verdict for the latest run.
latest = L.theLowdown('evaluationLatest.json')?.value

# How did we score on a specific run?
row = L.theLowdown("evaluation{#{runID}}.json")?.value

# Trend across ONLY comparable runs (same prompt set). Preferred.
series = L.theLowdown("evaluationsByPromptHash{#{latest.eval_prompts_hash}}.jsonl")?.value ? []
```

Note: `evaluationHistory.jsonl` returns ALL rows regardless of
comparability and is mostly useful for "find me runs to inspect."
For numeric reasoning, use `evaluationsByPromptHash{...}`.

Direct SQL via the meta layer is also fine when richer queries are
needed — `runtime.sqlite` is a normal sqlite file.

## Cross-table join: `trainingHistoryJoinEval.jsonl`

The two tables that matter for tuning use **different `run_id` keys**:
`evaluations.run_id` is the runner's UUID; `lora_training_runs.run_id`
is the training subprocess's `lora-<startedAt>` string. They can't be
joined by key, so the join is **temporal**: for each evaluation, attach
the most recent `lora_training_runs` row whose `finished_at <=
judged_at` (excluding any training still running).

Semantics this gives you:

- One eval → one training (the one in effect at scoring time).
- Multiple evals → same training when you re-eval without retraining
  (correct — the training IS what was in effect).
- Eval before any training exists → all `training_*` columns NULL
  (correct — there was no training to attribute to).
- Training in progress when eval ran → ignored (the `finished_at IS NOT
  NULL` filter excludes it; the previous completed training attaches
  instead, which matches what the adapter on disk actually was).

The training columns returned (all prefixed `training_`):
`run_id`, `started_at`, `finished_at`, `status`, `model_dir`,
`adapter_path`, `train_rows_count`, `valid_rows_count`,
`test_rows_count`, `checkpoint_path`.

The eval's `hyperparams_json` already contains the LoRA settings the
training subprocess ran with (snapshotted at eval time from
`params/run_lora_train_ite.yaml`), so the join is primarily for
training-side observables — row counts, checkpoint path, exact
training start/finish times — not for re-discovering the hyperparams.

Suggested use:

```coffee
joined = L.theLowdown('trainingHistoryJoinEval.jsonl')?.value ? []
# Group evals by training_run_id to see "this adapter scored X, Y, Z over time."
byTraining = {}
for r in joined
  continue unless r.training_run_id?
  (byTraining[r.training_run_id] ?= []).push r
```

## Known caveats

- `voice_cosine_*` columns are `NULL` when `fallback_used = 1`. Always
  check `fallback_used` before reasoning about voice signals.
- Score columns assume two variant names (`base`, `with_adapter`); a
  recipe that ablates differently would need schema changes here, not
  just in the producer. That's a deliberate tradeoff for SQL ergonomics
  — most queries want `score_base` and `score_with_adapter` as columns,
  not as JSON fields.
- `eval_prompts_hash` lets you filter for "comparable" runs but doesn't
  encode WHY they're comparable (same model? same corpus? same
  oracle prompt?). Cross-run comparisons need to verify those via
  `hyperparams_json` as well.
- The `hyperparams_json` snapshot is what each step actually saw at
  run time (after override merging) — not the canonical recipe. Two
  runs with the same `config/<recipe>.yaml` but different
  `override/<recipe>.yaml` will have different hyperparams here, which
  is the intent.
- Empty `evaluations` table on a fresh DB is normal; rows only appear
  after a first successful `judge_run_ite` against this runtime. If
  `fallback_used = 1` shows up consistently, the corpus probably needs
  oracle re-run to populate `kag_embeddings` first.
